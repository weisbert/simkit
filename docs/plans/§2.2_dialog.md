**Schema fields not on the form** (and why):
- `schema_version` вЂ” fixed at 1, written automatically (loader supports `_PVT_SUPPORTED_SCHEMA_VERSIONS '(1)`). No user input necessary.
- `_doc` вЂ” pure documentation. Auto-write a short `_doc` string saying "Created by simkit first-save dialog YYYY-MM-DD; edit by hand to add testbench aliases or change author."
- additional `testbench_aliases` entries вЂ” only one is offered on first save (the active cellview); users add more by editing the file later. Schema.md notes the file is "small, flat, and rarely hand-edited" (DECISIONS #13), so this is intentional minimum-viable.

**Default derivation: active cellview probe.** Mirror `_pvtCollAutoCapture` lines 351вЂ“364 in `pvtCollect.il`. Wrap each call in `errset` because the dialog can be invoked from a non-Maestro window:

```
(setq sess (errset (axlGetWindowSession) nil))
(setq sess (cond ((listp sess) (car sess)) (t nil)))
(when sess
  (setq lib  (errset (axlGetSessionLibName  sess) nil)) ...)
```

Failure to detect a session is **not an error** вЂ” just suppress the alias checkbox and fall back to cwd-basename for `project`.

## 3. API design

```skill
(procedure (pvtProjectFirstSaveDialog @key (defaults nil) (savePath nil))
  "Modal first-save bootstrap form. Returns the absolute path of the
   newly-written .pvtproject on Save, or nil on Cancel.
   ?defaults : optional alist of pre-fill values; tested fields are
              'project, 'dbRoot, 'author, 'aliasFor, 'aliasName.
   ?savePath : optional absolute target path; default <cwd>/.pvtproject.")
```

**Why this signature:**
- Returns `string|nil`, *not* a `pvtError` result вЂ” symmetric with `pvtFindPvtProject` which the loader treats the same way (see `pvtProject.il:327вЂ“329`). Loader code that already handles `pvtFindPvtProject` returning either path-or-nil reuses for the dialog return.
- `?defaults` lets a future caller (e.g. the В§5 `pvt` CLI, or a wrapper that wants to seed the form differently) override the auto-detected values without introducing globals.
- `?savePath` is mostly for tests вЂ” production callers always rely on the cwd default.

**Caller context wiring.** The loader already invokes the dialog with no arguments at `pvtProject.il:339`, which is correct: defaults derive themselves from the live session and cwd. **No change needed to that line**. The `pvtLoadPvtProject` `?allowDialog t` flag (already present, line 305) is the public on/off switch вЂ” batch callers (e.g. CI) pass `?allowDialog nil`.

**Cancel semantics.** When the dialog returns nil, the loader falls through to the existing step 4 hard error (`pvtProject.il:344вЂ“352`). The error message "no .pvtproject found walking up from <cwd> and PVT_PROJECT is not set" is already correct for this case вЂ” no rewording needed. The user has just clicked Cancel; they get the original "no file found" error per the brief.

## 5. JSON-write strategy

**`pvtJson.il` already has a writer.** `pvtJsonEmit` (string return) and `pvtJsonEmitToPort` (streams to an open port) are public, on lines 753 and 764. They:
- Walk the `_PVT_JSON_KEYS_PROP` sidecar for objects, so insertion order is preserved (line 862).
- Handle the schema's value types: strings (escape-aware), integers, the `PVT_JSON_NULL` sentinel, nested objects.
- Already in production use by `_pvtCollWriteRunJson` (`pvtCollect.il:892`).

**Reuse this. Do not write a new emitter.** Build a `makeTable 'pvtProjectFile PVT_JSON_NULL`, populate keys in schema-order, set the `_PVT_JSON_KEYS_PROP` sidecar, and call `pvtJsonEmitToPort`. This is the pattern `_pvtCollAutoCapture` uses at lines 383вЂ“404 for the run-meta table.

**Build pattern:**

```
(setq tab (makeTable 'pvtProjectFile PVT_JSON_NULL))
(setq keys (list "_doc" "project" "dbRoot"))                  ; required
(setarray tab "_doc"   docString)
(setarray tab "project" projectName)
(setarray tab "dbRoot"  dbRootValue)
(when authorValue
  (setarray tab "author" authorValue)
  (setq keys (append keys (list "author"))))
(when aliasObj                                                 ; optional table
  (setarray tab "testbench_aliases" aliasObj)
  (setq keys (append keys (list "testbench_aliases"))))
(setarray tab "schema_version" 1)
(setq keys (append keys (list "schema_version")))
(setarray tab _PVT_JSON_KEYS_PROP keys)
```

The nested `testbench_aliases` object is built the same way (a separate `makeTable`, with its own one-element sidecar key list).

**Write path** (mirror `_pvtCollWriteRunJson`):
```
(setq port (outfile path "w"))
(unwindProtect (pvtJsonEmitToPort tab port) (close port))
```
Wrap in `errset` so partial writes surface as a `pvtErr 'pvt_collect_io`-style error (re-categorize as `'pvt_dialog_io` to keep call-site context distinct).

**Round-trip self-check.** After the write, immediately call `pvtParsePvtProject` on the new file. If it returns `:err`, delete the file and return nil from the dialog (treat as Cancel) so the loader's hard-fail path runs with the original "no file" message, not a confusing "validation failed on a file you just wrote" message. This catches schema drift between dialog and loader at the earliest possible moment.

## 6. Default location

**Write to `<cwd>/.pvtproject`.** Justification:
- Schema.md В§1: "conventionally placed at the root of a circuit project tree."
- Loader walks **upward** from cwd (`pvtFindPvtProject`); writing into cwd guarantees the very next `pvtLoadPvtProject` call finds it.
- Mirrors how `git init` works: drops `.git/` in cwd; user is expected to invoke from the project root.
- `~/.pvtproject` is a *bad* default вЂ” it would shadow every project on the machine because the walker also matches an exact `$HOME` match before climbing past it.

**Editable in the form.** A power user with cwd at a subdirectory may want to write the file one level up. Show the path as a string field with the cwd-default pre-filled; on Save, validate that the parent directory exists and the file does not (refuse to overwrite вЂ” silently overwriting an existing `.pvtproject` from a dialog is too dangerous).

**Path simplification:** pipe through `simplifyFilename` before writing and before returning. The loader's step-4 path already simplifies, so consistency matters (`pvtProject.il:344вЂ“347`).

## 7. Test strategy

SKILL forms cannot be unit-tested without a live Virtuoso UI: `hiDisplayForm` blocks on user interaction, and the form-builder calls error out under `nograph` if invoked at the wrong time. Strategy is to push everything testable behind a pure boundary.

### Tier-1 (offline, mock-free) вЂ” runs in `runTests.il` headless

Split the dialog into three layers:

1. **Defaults derivation**: pure function `_pvtDlgDeriveDefaults(?cwd ?session)` that takes a cwd string and an optional faked session-tuple and returns an alist `((project ...) (dbRoot ...) (author ...) (aliasFor ...))`. No `hi*` calls. Fully testable.
2. **Field validators**: pure functions per field (`_pvtDlgValidateProject`, `_pvtDlgValidateDbRoot`, `_pvtDlgValidateSavePath`). Each takes a string and returns `pvtOk | pvtErr 'pvt_dialog_field`. These mirror the loader's validators but on raw string input (loader operates on parsed JSON values).
3. **JSON-write of finalized struct**: pure function `_pvtDlgWriteFile(?values ?targetPath)` that takes an alist and a path, builds the table, and writes via `pvtJsonEmitToPort`. Round-trippable through the existing `pvtParsePvtProject`.

Tier-1 test file: `skill/tests/testPvtProjectDialog.il`. Concrete test names:

- `dialog defaults: cwd basename used when no session detected`
- `dialog defaults: lib name normalized to project regex`
- `dialog defaults: $USER is the author default`
- `dialog defaults: invalid lib name falls back to cwd basename`
- `dialog validate project: rejects empty`
- `dialog validate project: rejects uppercase`
- `dialog validate project: rejects spaces`
- `dialog validate project: accepts hyphen + underscore + digit mix`
- `dialog validate dbRoot: rejects empty`
- `dialog validate dbRoot: accepts ./relative`
- `dialog validate dbRoot: accepts /absolute`
- `dialog validate dbRoot: accepts ~/home`
- `dialog validate savePath: rejects existing file (no-overwrite)`
- `dialog validate savePath: rejects parent dir missing`
- `dialog validate savePath: requires .pvtproject filename`
- `dialog write: round-trips through pvtParsePvtProject (minimum fields)`
- `dialog write: round-trips with author set`
- `dialog write: round-trips with one testbench alias`
- `dialog write: emits schema_version as integer (NOT string "1")`
- `dialog write: omits author key entirely when nil (does not emit "author": null)`
- `dialog write: refuses to overwrite an existing file`
- `dialog write: cleans up partial file on emitter error`

Target: ~22 tests. Brings total Tier-1 from 168 to ~190.

### Tier-2 (live Virtuoso UI, manual)

Documented in `skill/tests/tier2/scenarios.md` (per the В§3 testing pattern from TODO.md). Scenarios:

1. **Happy path**: cwd at a fresh dir with no `.pvtproject` anywhere up the tree. Open Maestro; `PvtSave` triggers dialog; fill minimum fields; click Save. Verify `.pvtproject` written; verify subsequent `PvtSave` succeeds without dialog.
2. **Cancel path**: same setup; click Cancel. Verify caller gets `:err 'pvt_notFound` with the existing message.
3. **Validation feedback**: enter invalid project name (e.g. `My LDO`); click Save. Verify form does not close and shows a validation error.
4. **Re-entrancy**: click Save twice quickly. Verify second click is no-op (form already disposed).
5. **Headless suppression**: run `virtuoso -nograph -replay` with `PvtSave` invoked. Verify dialog does NOT pop and caller gets `:err 'pvt_notFound`.

Tier-2 verification is a hard prerequisite per the standing rule вЂ” Decision #14 made `boundp`-gating + skillbridge probing canonical; same applies here.

## 9. Risk inventory

1. **Modal callback semantics under classic SKILL.** `hiCreateAppForm`'s `?callback` is invoked when the user clicks OK. Open question to verify via skillbridge BEFORE coding: does the callback run synchronously inside `hiDisplayForm` (so we can collect results from form fields before `hiDisplayForm` returns), or asynchronously (so we need a global flag and event-loop yielding)? If the latter, the dialog needs to use `hiFormDone` polling rather than direct return-by-callback. Probe form to verify on `/tmp/skill-server-default.sock`.

2. **`hiInBatchMode` not reliably defined.** The loader already side-steps this by checking `(getShellEnvVar "DISPLAY")` instead (see comment `pvtProject.il:333вЂ“334`). Carry the same convention; do **not** introduce `hiInBatchMode` as a new dependency.

3. **Re-entrancy: user double-clicks Save.** Both clicks fire the callback; the second one tries to write to a file that now exists. The "refuse to overwrite an existing file" validation in В§6 is the safety net вЂ” the second write fails, but by then the first has already returned the path. Need a module-level `_pvtDlgInFlight` flag that the callback sets on entry and clears on exit, OR use `hiFormCancel` to dispose the form before doing the write so re-clicking has nothing to act on. Recommendation: dispose first.

4. **Error during write leaves a corrupt `.pvtproject`.** If `pvtJsonEmitToPort` raises mid-stream, the file exists but is invalid JSON and the loader will reject it on the next `pvtParsePvtProject` call вЂ” leaving the user permanently broken until they manually delete it. Guard with `unwindProtect` + `errset`, and if the emitter errors, **`deleteFile`** the partial output before returning nil. Mirror `_pvtCollWriteRunJson`'s pattern but add the cleanup branch (the collector doesn't bother because partial run dumps are not on the loader's critical path; first-save IS).

5. **Defaults from `axlGetSessionLibName` may be in unexpected case / contain disallowed chars.** Real lib names commonly use uppercase (`MY_LIB`, see `pvtproject.example.json`). The project regex `^[a-z0-9_-]+$` rejects them. Lowercase + replace non-matching chars with `_`, but show the *raw* lib name as a comment under the field so the user understands why it differs. If after normalization the result is empty, fall back to cwd basename.

6. **JSON byte-representation NUL limitation (DECISIONS #15).** Theoretically the user could paste a NUL into an author or project field. Practically: project regex forbids it; dbRoot is a path so can't contain NUL on Unix; author is the only field exposed. The risk is essentially nil in practice but worth one defensive `pcreMatchp "^[^\x00]*$"` on the author field before writing. Note that the byte-LUT issue is for the parser side; the emitter side does not synthesize bytes (see `pvtJson.il:746`). So the only failure mode is "user types a NUL, parser later chokes when re-reading the file." Guard at write time, not at parse time.

7. **`hiDisplayForm` under `nograph`.** Even with the `DISPLAY`-env gate, a misconfigured Cadence install (DISPLAY exported but X server not actually reachable) will hang. Mitigation: caller's `?allowDialog nil` is the only foolproof escape. Document this in the dialog's docstring and in `README.md`: "First-save dialog respects DISPLAY but cannot detect a broken X server. Pass `?allowDialog nil` to skip in CI."

8. **Test loader-order sensitivity.** `runTests.il` currently loads pvtError в†’ pvtJson в†’ pvtProject в†’ pvtCollect. Adding `pvtProjectDialog.il` after `pvtProject.il` means tests for `pvtLoadPvtProject` will now see `(boundp 'pvtProjectFirstSaveDialog)` flip to true. This could change behaviour of existing tier-1 tests that assume the dialog is absent. **Verify**: every existing call to `pvtLoadPvtProject` in `testPvtProject.il` either passes `?allowDialog nil` or runs in an environment where `getShellEnvVar "DISPLAY"` is nil (it is, under nograph). Add an explicit `?allowDialog nil` audit sweep.

9. **Dialog is loaded but `axl*` calls fire from a non-Maestro window.** Each `axlGetSession*` call must be wrapped in `errset` (see `pvtCollect.il:351вЂ“355`) and the result destructured defensively (`(cond ((listp x) (car x)) (t nil))`). Skipping this will hard-error the dialog before it even displays.

10. **Re-running: dialog written file is invalid.** If a corner case sneaks through field validation but fails `pvtParsePvtProject` round-trip (the В§5 self-check), the dialog deletes the file and returns nil. The user sees "no .pvtproject found" вЂ” no signal that the dialog tried and failed. Mitigation: log the round-trip failure to the CIW with a `warn`-level message, including the pvtError category and message, before returning nil. Cost: one extra line; benefit: any future schema/validator drift is visible.
