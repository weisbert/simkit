# Schema Spec

**Schema version: 1** — frozen surface for Phase 1. Any breaking change requires bumping `schema_version` and appending a migration note to `DECISIONS.md`.

This document defines five things, in dependency order:

1. `.pvtproject` JSON format (project identity)
2. JSON dump format (what the collector SKILL writes per run)
3. DuckDB tables (what the Python ingester loads into)
4. Run-directory layout on disk
5. Versioning policy

Every non-obvious field cites the `DECISIONS.md` entry that drives it.

---

## 1. `.pvtproject` JSON format

A project is declared by a single JSON file, conventionally placed at the root of a circuit project tree. It is discovered by walking up from cwd (see DECISIONS #6). Format is strict JSON (no comments, no trailing commas) — see DECISIONS #13 for why JSON over YAML/TOML.

| Field | Type | Required | Default | Validation / notes |
|---|---|---|---|---|
| `project` | str | yes | — | Project identifier. Must match `^[a-z0-9_-]+$`. Used as `project_id` in every dump. (#6) |
| `dbRoot` | path | yes | — | Root directory for the DuckDB file and all run dumps. If relative, resolved from the `.pvtproject` file's own directory. If absolute, used as-is. (#2) |
| `author` | str | no | value of `$USER` at dump time | Recorded into each run. Captured at dump time and stored; never looked up live at query time. (#10) |
| `testbench_aliases` | object[str → str] | no | `{}` | Maps `lib/cell/view` → human-readable alias. Alias values must be unique within the project; duplicate aliases = load error. (#7) |
| `schema_version` | int | no | `1` | Pins the `.pvtproject` file format version. Separate from the JSON-dump `schema_version`. |
| `_doc` | str \| object | no | — | Reserved for free-form human documentation (JSON has no comments). Loader ignores it; never validated. |

**Unknown-key policy.** Top-level keys not listed above (except `_doc`) are warned (not errored) when the loader parses the file. Forward-compat: users may see newer templates carrying fields the current simkit doesn't yet understand. Keys starting with `_` are reserved for user documentation and are silently ignored.

**Example.** See `config/pvtproject.example.json`.

## 2. JSON dump format (per run)

The collector SKILL writes one JSON file per dump: `<dbRoot>/runs/<run_id>/run.json`.

Top-level structure:

```
{
  "schema_version": 1,
  "run": { ... run-level metadata ... },
  "results": [ ... per-result records ... ],
  "artifacts": [ ... initial artifact records written at dump time ... ]
}
```

### 2.1 Run-level metadata (`run` object)

| Field | Type | Notes |
|---|---|---|
| `run_id` | str (UUIDv4) | Generated by the collector. Uniqueness matters; monotonicity does not (the DB already has `timestamp`). |
| `project_id` | str | Copied from `.pvtproject:project`. (#6) |
| `testbench_id` | str | Maestro active setup's cellView path in the form `lib/cell/view`. (#7) |
| `testbench_alias` | str \| null | Resolved from `testbench_aliases` at dump time. `null` if no alias is defined for this `testbench_id`. (#7) |
| `timestamp` | str | ISO 8601 with offset, e.g. `2026-04-22T14:32:15+08:00`. Local wall time is preserved for a single-author offline tool. |
| `author` | str | From `.pvtproject:author` if set, otherwise `$USER` captured at dump time. (#10) |
| `label` | str \| null | `null` at dump time. Upgraded via `pvt label` to promote the run to a slice. (#11) |
| `note` | str \| null | Free-form dump-time note supplied by the user. (#10) |
| `netlist_path` | str | Path to the copied simulated netlist, relative to the run's dump dir. File is `input.scs`. (#8) |
| `history_name` | str | Maestro history name that the dump was taken from. Available via the POC's ADE session calls. |

All fields are required unless marked `\| null`.

### 2.2 Per-result records (`results` array)

Each entry is one row of the Maestro history's cross-product of points × corners × tests × outputs.

| Field | Type | Notes |
|---|---|---|
| `point` | int | Parameter-sweep point index. |
| `corner` | str | Corner name, e.g. `TT`, `WC_VDD`. |
| `test` | str | Test name. |
| `output` | str | Measurement name. May be the sentinel `__sim_status__` (see below). |
| `value` | number \| str \| null | Numeric if the measurement produced one, string for categorical outputs, `null` when `status != ok`. |
| `status` | str enum | Closed set: `ok` \| `failed` \| `running` \| `no_convergence`. Any other value is a hard error at ingest. |
| `sweep` | object | Sweep parameter conditions, e.g. `{"W": "10u", "L": "100n"}`. Keys are strings; values are strings or numbers. |
| `corner_vars` | object | Parsed corner variables, e.g. `{"vdd": 1.8, "model": "nch\|pch"}`. Same shape as `sweep`. |
| `test_note` | str \| null | Maestro per-test notebook text, if any. Populated by the collector (see TODO §3); spec pins that the field exists. (#10) |

**Sim-status sentinel rows.** The POC emits rows with `output == "__sim_status__"` and `value == null` to carry test-level status when no per-output result exists (e.g. a test failed before any measurement produced output). v1 preserves this convention. The ingester treats sentinel rows as test-level status rows; numeric-output queries must filter them out.

### 2.3 Artifact records (`artifacts` array)

Artifacts attached at dump time (auto-captured waveforms, screenshots, logs) appear here. Post-hoc attachments (via `pvt attach`) are added directly to the DB and do not retroactively modify `run.json`.

| Field | Type | Notes |
|---|---|---|
| `type` | str enum | Closed set in v1: `waveform` \| `results_table` \| `sim_log` \| `schematic` \| `netlist_diff` \| `image` \| `pdf` \| `other`. Adding a type requires a `schema_version` bump. |
| `relative_path` | str | Path relative to the run's dump dir, typically under `artifacts/`. |
| `description` | str | Short human-readable note. |
| `source` | str enum | `auto` (written by collector) \| `manual` (added later via `pvt attach`). (#9) |
| `created_at` | str | ISO 8601 with offset. For `auto`, equal to the run's `timestamp`; for `manual`, the attach time. |

## 3. DuckDB tables

The Python ingester reads `run.json` files and loads them into a single DuckDB file at `<dbRoot>/simkit.duckdb`.

```
runs(
  run_id          VARCHAR PRIMARY KEY,
  project_id      VARCHAR NOT NULL,
  testbench_id    VARCHAR NOT NULL,
  testbench_alias VARCHAR,
  timestamp       TIMESTAMPTZ NOT NULL,
  author          VARCHAR NOT NULL,
  label           VARCHAR,
  note            VARCHAR,
  netlist_path    VARCHAR NOT NULL,
  history_name    VARCHAR NOT NULL,
  schema_version  INTEGER NOT NULL,
  ingested_at     TIMESTAMPTZ NOT NULL
)

results(
  run_id          VARCHAR NOT NULL REFERENCES runs(run_id),
  point           INTEGER NOT NULL,
  corner          VARCHAR NOT NULL,
  test            VARCHAR NOT NULL,
  output          VARCHAR NOT NULL,
  value_num       DOUBLE,
  value_str       VARCHAR,
  status          VARCHAR NOT NULL,
  sweep           JSON NOT NULL,
  corner_vars     JSON NOT NULL,
  test_note       VARCHAR
)

artifacts(
  run_id          VARCHAR NOT NULL REFERENCES runs(run_id),
  type            VARCHAR NOT NULL,
  relative_path   VARCHAR NOT NULL,
  description     VARCHAR,
  source          VARCHAR NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL
)
```

**Column notes.**

- `runs.label`: `NULL` = draft run (GC-eligible); non-NULL = slice (retained permanently). (#11)
- `runs.ingested_at` vs. `runs.timestamp`: `timestamp` is when the sim was dumped from Virtuoso; `ingested_at` is when Python loaded the JSON into DuckDB. Both are needed — one for history reasoning, one for "what did I load today."
- `results.value_num` / `results.value_str`: the JSON `value` union is split on load so numeric queries (aggregations, thresholds, cross-corner min/max) stay SQL-native without JSON unwrap. At most one of the two is non-NULL per row; both NULL when `status != ok`.
- `results.output == '__sim_status__'`: sentinel rows. Numeric queries must filter `WHERE output <> '__sim_status__'`.
- `artifacts`: no primary key declared. `(run_id, relative_path)` is the natural uniqueness pair; enforce at ingest time, not via DDL (DuckDB index ergonomics + ingester already owns idempotency).

**Indexes and slice views.** Deferred to the ingester/query-layer task. Both are loader-time concerns; v1 schema only pins the columns.

## 4. Run-directory layout

Flat by `run_id`. UUIDv4 already disambiguates; date-bucketing would add path complexity to every `pvt attach` / `pvt diff` invocation for no query benefit (the DB has `timestamp`).

```
<dbRoot>/
├── simkit.duckdb                      (DB file; created by the ingester)
└── runs/
    └── <run_id>/
        ├── run.json                   (the dump; §2)
        ├── input.scs                  (copied simulated netlist; #8)
        └── artifacts/
            ├── <auto>.png             (collector-saved)
            └── <manual>.pdf           (post-hoc attached)
```

All paths inside `run.json` (`netlist_path`, `artifacts[].relative_path`) are relative to `<dbRoot>/runs/<run_id>/`. This keeps the run directory self-contained and relocatable.

## 5. Versioning policy

- `schema_version = 1` is permanent for the v1 surface. Both the JSON dump and the DuckDB tables share the same version number in v1.
- **Breaking changes** (renaming a field, tightening a type, removing a field) require bumping `schema_version` and writing a migration note in `DECISIONS.md`.
- **Additive JSON changes** (new optional field the ingester can tolerate as missing) are allowed within v1.
- **DuckDB DDL changes** always require a version bump, even additive ones. DB migrations are not free — making each one a deliberate act is worth the ceremony.

Every ingested run carries its own `schema_version` (in JSON and in the `runs` row). When a future version ships, the ingester dispatches on this field.
