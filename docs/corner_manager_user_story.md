# SimKit Corner Manager — 用户故事

> 记录日期：2026-05-20
> 来源：用户（RFIC designer）口述需求，原文保留。本文档用于需求打磨阶段讨论，**不含实现设计**。

---

## 用户故事原文

作为 RFIC designer，我现在开始了一个复杂的 LO 系统设计，现在我开始设计 corner。这个 LO
系统内部单个模块并不复杂，就是不同通路 buf 和 DIV 的组合，复杂的是不同模式的寄存器控制。
例如假如说，有 7 个不同的工作模式，例如 BT_2G_RX、BT_2G_TX……

这位 Designer 在设计 corner 时遇到了如下的烦恼：

### 痛点 a：PVT corner 要按模式重复设计 N 次

类似 PVT corner 要重复设计 7 次。用寄存器配置为 BT_2G_RX 举例，对于不同的仿真需求，用户
设置了多列的 PVT 情况，如下：

- **BT_2G_RX_TT**：一列 corner 就是 TT，55℃，0.9V
- **BT_2G_RX_SS_1**：用户自定义的一个 SS corner 的情况，例如 SS，125℃，0.85V
- **BT_2G_RX_FF_1**：用户自定义的一个 FF corner 的情况，例如 FF，-40℃，0.95V
- **BT_2G_RX_PVT_16**：用户自定义的一个 corner 聚合
- **BT_2G_RX_PVT_45**：用户自定义的另外一个 corner 聚合
- ……

这种设计 corner 聚合其实是某种固定的设置，当用户设定完了 BT_2G_RX 这个模式后，还要手动继续
把剩下的 6 种模式按照同样的模式设定，非常浪费时间。

用户也想提醒，corner 聚合模板也不能是唯一的，例如 BT_2G_RX 需要定义这么多的 corner，但是对
于另外一个 corner Test_leakage，可能就只需要 TT、SS_1、FF_1。

### 痛点 b：改某个模式的寄存器配置时无法全局更改

当用户想改变 BT_2G_RX 的某个寄存器配置时无法全局更改。例如用户在设计中，电路结构改变了，某个
寄存器 d_en_dummy 对于 BT_2G_RX 模式需要从 1 改为 0，用户也只能自己手动把剩下的 BT_2G_RX
d_en_dummy 寄存器更改。

### 痛点 c：给某个模式添加变体测试，要把 corner 需求重编一次

当用户想给某个模式添加变体测试。例如用户之前配置的 BT_2G_RX 这个都是用来跑 trans 的，此时
用户想跑 PSS，发现电路有个 DIV12 开着，PSS 不收敛，于是想做一个 corner 把相关 DIV12 的寄存器
都关掉，于是用户新建了一个变体 corner BT_2G_RX_PN_TT。此时用户又要又一次把（a）中的 corner
需求重新编辑一次。

### 痛点 d：模式多、corner 多，难以管理 —— 希望有“组合”概念

由于模式太多，corner 也太多，用户不好管理。例如，用户想跑全模式的 TT corner，需要手动勾选
BT_2G_RX_TT、BT_2G_TX……；然后用户又想只跑 BT_2G_RX_TT、BT_2G_RX_SS_1、BT_2G_RX_FF_1，
需要先把之前跑全模式的 TT corner 取消掉，然后再勾选想要的 corner。

此时用户就在想，要是有个类似“组合”的管理概念就好了，例如用户可以命名一个组合为 “All_Mode_TT”，
里面全是 7 个模式的 TT corner；然后用户再命名第二个组合为 “BT_2G_RX_Fast_check”，这个组合
里面是 BT_2G_RX_TT、BT_2G_RX_SS_1、BT_2G_RX_FF_1；这样用户就可以选择组合之间的切换。

### 痛点 e：corner 变量（行）太多时不方便查找 —— 希望有变量名筛选

当 corner 的变量太多时，不方便查找。用户发现当 corner 里面的控制信号特别多时，例如 >100 行了，
Cadence 的 corner manager 里面没有可以筛选变量名的功能，用户只能自己用肉眼找，很不方便。要是
在 corner 表格的顶部有个地方，可以输入字符，用户可以选择（and、or、一些常见的搜索匹配方法）他
想专门看的变量就好了。

### 痛点 f：corner 列太多时 —— 保留按名字筛选，并增加按集合筛选

当 corner 的列数太多时，Cadence 的 corner manager 里面提供了根据名字筛选 corner 的功能。如果
要自己做 corner manager 的脚本的话，用户希望这个功能可以保留，同时增加（d）中所说的按照集合筛选。

### 痛点 g：原生 corner 编辑器无法拖动变量的行

Cadence 原生的 corner 编辑器无法拖动变量的行数。用户在编辑完 corner 内的变量后，想再添加一个新
的变量 LDO_VSET，并且想放在第三行，因为用户喜欢把 LDO 相关的变量放在前几行。但是 Cadence 只能
放在最后，没有办法拖上去。

## 讨论澄清（2026-05-20）

针对初次讨论的疑问，用户澄清如下：

1. **"corner 聚合"（PVT_16 / PVT_45）是什么** —— 它也是 Cadence corner manager 里的
   一列 corner，但这一列里包含 16 个或 45 个仿真点。以 PVT_45 为例：
   `(TT, SS, FF, FS, SF) × (55℃, 125℃, -40℃) × (0.9V, 0.85V, 0.95V)`。
   用户用 Cadence 时按需求勾选不同 corner：只想看 TT 就只勾 `BT_2G_RX_TT`；想收集完整
   或部分完整数据时就用 `PVT_45` / `PVT_16`。
   → 即"聚合"是**单列、内含多个 point-sweep 点**；和痛点 d 的"组合/集合"（跨模式的
   corner 勾选清单）是不同层级的概念。

2. **变体的作用范围** —— 变体会生出一个"BT_2G_RX 的 PSS 变体模式"，这个变体模式自己
   也能套 PVT 模板（PN_TT、PN_SS_1…）。即变体是一等公民，可被模板叉乘。

3. **模板套用方式** —— 持续绑定；同时需要一个**解绑**功能。

## 补充用户故事

### 痛点 h：添加 VCO 类型的 PVT 的麻烦

> 相关背景：`docs/phase2_pvt_union_spec.md` 记录的 VCO LO 2026-05-11 motivating case。

VCO 的一个 PVT，里面的 corner-var **不是各自独立的** —— 有些 var 必须**绑定在一起**变化：

- CT-tuning（粗调）跟着**工艺角**走：TT→CT=100，FF→CT=88，SS→CT=120，FS→CT=100，
  SF→CT=100
- `.s5p` 电感模型文件跟着**温度**走：55℃→L_55.s5p，125℃→L_125.s5p，-40℃→L_n40.s5p

如果像 point-sweep 那样把每个 var 都当成**独立轴**做笛卡尔积，会变成：

```
(TT, FF, SS, FS, SF)
  × (常压, 高压, 低压)
  × (55℃, 125℃, -40℃)
  × ("L_55.s5p", "L_125.s5p", "L_n40.s5p")
  × (CT=88, CT=100, CT=120)
= 5 × 3 × 3 × 3 × 3 = 405 个组合
```

绝大部分是无意义的乱配（例如 TT 配 CT=88、55℃ 配 L_125.s5p）。

用户真正想要的，是把**绑定的 var 捆成一个"复合轴"**再做笛卡尔积：

```
[TT(CT=100), FF(CT=88), SS(CT=120), FS(CT=100), SF(CT=100)]
  × [常压, 高压, 低压]
  × [55℃("L_55.s5p"), 125℃("L_125.s5p"), -40℃("L_n40.s5p")]
= 5 × 3 × 3 = 45 个 corner
```

> VCO 痛点的真正原因是 **corner-var 之间存在绑定 / 相关关系**：point-sweep 只能对**独立轴**
> 做笛卡尔积，无法表达"某个 var 要跟着另一个 var 一起走"。为了表达这种绑定，用户只能手工把
> corner 一列列拆开、逐列把相关的值钉死。
>
> 待澄清：Maestro 的 point-sweep 能否扫标量值，用户尚未尝试过，不确定。
